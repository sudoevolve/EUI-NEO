# 状态模型

EUI-NEO 的状态模型优先保持单向数据流：页面或业务层持有业务状态，组件声明当前值，组件交互时只回调 next value。组件内部可以持有纯交互状态，Runtime 可以持有高频视觉状态，但它们都不应该变成业务真相。

这个口径参考两个成熟模型：

- Flutter：控件拿 `value`，交互时触发 changed callback，父级更新状态后下一次 build 再把新值传回控件。
- QML：对象暴露 property，属性变化通过 `on<Property>Changed` 这类 handler 表达。

EUI-NEO 采用更适合 C++ 链式 builder 的短命名：值叫 `value/open/checked/selected`，变化叫 `onChange/onOpenChange`。规则要简单，不为每个组件发明一套特殊名字。

## 状态分层

| 层级 | 归属 | API | 适用场景 |
| --- | --- | --- | --- |
| 业务状态 | 页面、业务对象、上层 app | 普通 C++ 变量、结构体、`Signal<T>` | 表单值、选中项、弹层开关、滚动 offset、应用数据 |
| 组件私有状态 | 当前 `Ui` / loader scope | `ui.state<T>(id)` | 输入框光标、拖拽草稿、picker draft、动画辅助计时 |
| Runtime 交互状态 | Runtime 实例表 | `scrollState(...)`、`sliderState(...)`、`visualStateFrom(...)` | 滚动、滑块拖动、hover/pressed 缩放、pointer-follow transform |
| 渲染缓存状态 | Render backend / Runtime | framebuffer cache、primitive cache、text/image cache | 脏区重绘、静态内容复用、资源复用 |

这四层的依赖方向固定：页面声明目标状态，组件把目标状态写进 DSL，Runtime 根据 DSL 推进交互和动画，渲染后端同步 primitive。底层不反向拥有业务状态。

## 受控组件

通用组件默认是受控组件。页面传入当前值，组件通过回调请求页面更新：

```cpp
components::slider(ui, "volume")
    .value(volume)
    .onChange([&](float next) {
        volume = next;
    })
    .build();
```

标准命名：

| 状态语义 | 当前值 API | 变更 API | 示例 |
| --- | --- | --- | --- |
| 通用标量或对象 | `value(T)` | `onChange(T)` | input、slider、stepper、colorPicker |
| 布尔选择 | `checked(bool)` | `onChange(bool)` | checkbox、toggleSwitch |
| 选中项 | `selected(int/bool)` | `onChange(...)` | tabs、segmented、radio、dropdown |
| 连续位置 | `index(float)` | `onChange(float)` | carousel |
| 可见性 | `open(bool)` / `visible(bool)` | `onOpenChange(bool)` / dismiss 回调 | dropdown、dialog、sidebar、contextMenu、toast |
| 滚动位置 | `offset(float)` | `onChange(float)` | scrollView |

`onChange` 的含义是“组件建议的新值”，不是“组件已经持有了新业务状态”。页面可以接受、修正或忽略这个值。

新组件 API 命名按这个优先级选择：

1. 能叫 `value` 就叫 `value`，对应 `onChange`。
2. 布尔开关叫 `checked`，对应 `onChange`。
3. 选项叫 `selected`，对应 `onChange`。
4. 可见弹层叫 `open`，对应 `onOpenChange`。
5. 临时反馈叫 `visible`，对应 `onDismiss` / `onAutoDismiss`。
6. 滚动叫 `offset`，对应 `onChange`。
7. 只有领域语义明显强于 `value` 时才新增名字，例如 `date(...)`、`time(...)`、`items(...)`。

不要新增 `onValueChanged`、`onSelectedChanged`、`onCheckedChanged` 这类长名字。它们更接近 QML，但在当前 C++ builder 里会让调用变啰嗦。EUI-NEO 对外统一用较短的 `onChange`；只有 `open` 因为经常和选择、提交等事件并存，所以使用明确的 `onOpenChange`。

## 弹层开关

可开关弹层统一优先使用：

```cpp
.open(open)
.onOpenChange([&](bool next) {
    open = next;
})
```

`open(bool)` 组件只提供 `onOpenChange(bool)` 表达开关变化。关闭按钮、遮罩点击、菜单 dismiss 都统一回调 `onOpenChange(false)`。

`visible(bool)` 用于 toast 这类短暂反馈，不一定表示可交互弹层。它可以继续搭配 `onAutoDismiss()` / `onDismiss()`。

## 组件私有状态

组件内部需要保存纯交互状态时使用 `ui.state<T>(id)`：

```cpp
struct DragState {
    bool dragging = false;
    float startX = 0.0f;
};

DragState& state = ui.state<DragState>(id + ".drag");
```

规则：

- `ui.state<T>` 只保存组件私有状态，不保存页面业务真相。
- 状态 id 必须稳定，通常使用组件 id 或 `id + ".name"`。
- 不要用组件级 `static unordered_map<id, state>` 管理状态。
- Loader 会把状态归入自己的 scope；`destroyOnHide()` 释放，`keepAlive()` 保留。
- draft 状态可以暂存用户正在编辑但尚未提交的值，例如 picker 面板里的临时日期。

## Runtime 状态绑定

高频交互优先走 Runtime 状态绑定，避免每次 hover、drag、scroll 都触发页面 compose：

```cpp
ui.stack("button")
    .visualStateFrom("button.surface", 0.965f)
    .build();

ui.stack("viewport")
    .scrollState("viewport", scrollY, maxOffset, 48.0f)
    .onScrollOffsetChanged([&](float next) {
        scrollY = next;
    })
    .build();
```

命名规则：

- `ui.state<T>(id)` 表示组件私有实例状态。
- `scrollState(...)` / `sliderState(...)` 表示 Runtime 交互状态源。
- 组件 builder 不应再新增模糊的 `.state(...)`；如果需要传 Runtime state id，使用类似 `scrollStateId(...)` 的明确名字。
- `visualStateFrom(...)`、`scrollContentFrom(...)`、`scrollThumbFrom(...)` 等 `From` API 表示“从某个 Runtime 状态源读取派生视觉状态”。

## Signal 双向绑定

`Signal<T>` 是受控组件的显式双向绑定语法糖，不是另一套状态模型：

```cpp
eui::Signal<float> volume{0.62f};

components::slider(ui, "volume")
    .bind(volume)
    .build();
```

等价于：

```cpp
components::slider(ui, "volume")
    .value(volume.get())
    .onChange([&](float next) {
        volume.set(next);
    })
    .build();
```

这样可以保持 API 一致：简单页面继续用普通变量和手写回调，大页面可以用 signal 减少样板代码，底层 Runtime 绑定仍负责高频视觉和输入状态。

常用绑定方法：

| 组件状态 | Signal API | 等价受控写法 |
| --- | --- | --- |
| `value(...)` | `bind(signal)` | `value(signal.get()).onChange(...)` |
| `checked(...)` | `bind(signal)` | `checked(signal.get()).onChange(...)` |
| `selected(...)` | `bind(signal)` | `selected(signal.get()).onChange(...)` |
| `index(...)` | `bind(signal)` | `index(signal.get()).onChange(...)` |
| `offset(...)` | `bind(signal)` | `offset(signal.get()).onChange(...)` |
| `open(...)` | `bindOpen(signal)` | `open(signal.get()).onOpenChange(...)` |
| `visible(...)` | `bindVisible(signal)` | `visible(signal.get()).onDismiss(...)` |

如果交互回调里还需要做额外副作用，例如关闭其它弹层或写日志，可以继续使用显式 `value/open + onChange/onOpenChange` 写法。不要在同一个组件上同时写 `bind(...)` 和同类 `onChange(...)`，后写的回调会覆盖前面的绑定。
